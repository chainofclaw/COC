## Initial success (~05:06–05:08Z, ~3 min)

5-validator BFT was working. Logs show `BFT round finalized` every 2-3s:
```
05:08:34Z h=71399 finalized
05:08:36Z h=71400 finalized
05:08:42Z h=71401 finalized
05:08:49Z h=71402 finalized
```

Vote pattern: `prepareVotes:4, commitVotes:3` — only 4 of 5 validators voted in prepare phase per round (probably the rotation proposer doesn't pre-vote). With relaxedQuorum threshold = (160*2)/3 = 106 stake, 4 × 32 = 128 stake satisfied quorum.

## Failure on f=1 chaos (05:08:45Z stop)

When server-3 (anvil-2 in our remapped topology) stopped, **chain froze immediately**:
- Last round h=71402 closed with `prepareVotes:3, commitVotes:0, buffered:1`
- No round started for h=71403
- Even though 4 alive validators (server-1+server-2+anchor-1+anchor-2) = 128 stake should meet quorum

This means N=5 with our config has the **same single-fault fragility as N=3**, despite more validators. Hypothesis: BFT round mechanism / proposer rotation / cached `lastProposed` state hangs on dead validator's slot regardless of N.

## Failure on rollback (05:16:50Z onwards)

After rollback to N=3, **chain still does not advance**:
- Configs restored correctly: `validators=3` (anvil-0/1/2), `enableBft=true` ✓
- Wire connections established between all 3 prod nodes ✓
- All peer handshakes show h=71402 ✓
- server-3 stuck at h=71401 (1 block behind, can't apply 71402 from peers)
- Server-1 BFT log shows no `BFT round started` after restart

This indicates persistent state corruption in leveldb (chain-state or BFT round state) preventing fresh round from initializing, even after configs reverted.

## Total chain downtime

From 05:08:49Z (last finalized) to current — **~17 min and counting**, expected to extend until H15 fallback (~05:31:57Z if it works) for **~22 min total**.

## Lessons learned

1. **Live N=5 deployment without prior dev-devnet validation is unsafe.** Should have run `bash scripts/start-devnet.sh 5` + chaos drill locally first. Memory `coc-validator-registry-attempt.md` warned about this exactly.
2. **Coordinated config swap on running BFT cluster is fragile.** Even with backups + simultaneous restart, the old chain head + BFT state interfere with new config.
3. **Rollback isn't always recovery.** Once BFT state is wedged, restoring config alone doesn't unstick the chain. Need to either wait for H15 (10 min) or wipe BFT state.
4. **Topology mapping must be inspected before deploying.** I assumed `server-1=anvil-2` based on the chaos test plan; the actual mapping was `server-1=anvil-0`. Caught and corrected before the first restart, but only by chance (apply script logged the existing nodeId).

## Path forward

1. Wait for H15 fallback (~05:31:57Z) — if the chain self-recovers, document and close N=5 attempt
2. If H15 doesn't fire or doesn't help, investigate forced recovery (wipe BFT state in leveldb)
3. Long term: build N=5 config in chainId 88780 R3.2 fresh chain, NOT in-place 18780
4. Add chaos drill in dev devnet as standard pre-prod validation
